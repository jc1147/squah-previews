SEO Audit Platform

Audit a site, plan the overhaul, prove it landed — mostly for free.

A deterministic, free-tier, on-page SEO system that turns a live website into an implementer-ready overhaul plan and then proves the plan actually shipped — without ever touching the live HTML. It’s built for a solo operator or small agency running on-page SEO across multiple client sites, organised in an agency → client → site dashboard. Scanning and verification cost $0; only the full audit spends money, and never a cent until you approve the exact pages.

Everything runs from one working directory — Code/SEO Audit/pipeline/ — as three commands. The slash command and its raw node equivalent do the same work; on this Windows box /seo-* auto-discovery doesn’t fire, so the CLI form below is the reliable entry point.

⏻

/seo-scan <url>

Crawls any URL with the free Screaming Frog CLI and ingests it into the central seo.db — identical to clicking “Run scan” in the dashboard. No audit, no SERP, no LLM.

$0 Needs: Screaming Frog CLI
⚙

/seo-engine <url>

The full executive on-page audit + overhaul plan as a multi-tab report: right target phrase per page, competitor blueprints, relevancy verdicts, meta rewrites, linking, schema, canonical, positioning. Runs in 3 Acts around an approve-before-spend gate.

Paid Needs: Screaming Frog + DataForSEO
✓

/seo-qa <plan> <rescan>

Proves what a plan actually changed (PLAN→ACTUAL) or where a site moved between two scans (BEFORE→AFTER). Hard-verifies title, meta, H1 and schema. Read-only, no DB writes — no crawler needed.

$0 Needs: two crawls in seo.db

Cost in one line. /seo-scan and /seo-qa are $0. Only /seo-engine spends — real DataForSEO SERP usage, hard-gated by --cap (default 0.50 USD) with a server-side $2.00 ceiling on top. The per-page AI step uses your own Claude Code session, not a billed API key.

node run.mjs https://example-client.com

# 2. Start the dashboard  ->  http://localhost:5505  (log in james / Default1)
node server.mjs

# 3. Verify a plan landed: diff plan crawl 13 against re-scan crawl 22 (free)
node qa-compare.mjs 13 22

Get going

Quick start (about 15 minutes)

Five steps from a clean machine to your first scan showing in the dashboard. Two of the three commands cost nothing — only /seo-engine ever spends money, and DataForSEO (step 3) is optional, so you can skip it and still run a full free scan. Everything launches from one working directory: Code/SEO Audit/pipeline/.

  1. Install Node.js 24

    The engine runs on the built-in node:sqlite module, so there is no npm install — zero third-party dependencies. Node 24 is recommended because it needs no flag. Node 22.5–23 also works but requires --experimental-sqlite. Install Node, then confirm the version: 

    node --version    # must be >= 22.5; 24.x needs no flag

    Minimum is 22.5.0. Anything older will fail to load node:sqlite. On 22.5–23 you would pass --experimental-sqlite on every command; Node 24 drops that requirement entirely.

  2. Install Screaming Frog (free) & accept the EULA

    Scanning is powered by the free Screaming Frog SEO Spider CLI. The free tier caps each crawl at 500 URLs (fine for most sites; large catalogs use the sitemap path covered in Workflows). Two things have to be true before a headless crawl will run:

    1. Download and install the free Screaming Frog SEO Spider.
    2. Launch the desktop app once and accept the EULA.
    3. Point the engine at the CLI executable so it can find it.
    EULA gotcha — do this or every crawl refuses to run

    Headless CLI crawls silently refuse to start until the EULA is accepted once in the desktop app. If your very first scan fails immediately, this is almost always the cause: open the Screaming Frog desktop app, accept the licence, close it, and re-run.

    Where to put it / how to point at it. The packaged team toolkit reads an SF_CLI environment variable, so you can install Screaming Frog anywhere and point SF_CLI at the .exe: 

    $env:SF_CLI = "C:\Program Files (x86)\Screaming Frog SEO Spider\ScreamingFrogSEOSpiderCli.exe"

    Live pipeline/ copy is different. The live tree hardcodes the CLI path with no SF_CLI override — it expects Screaming Frog at Code/SEO Audit/Screaming Frog SEO Spider/. So either install (or symlink) it into that folder for the live copy, or use the toolkit copy and set SF_CLI. The engine fails closed if the CLI cannot be found.

  3. Optional — DataForSEO (only for /seo-engine)

    Paid  Skippable

    DataForSEO is the only paid external API, and it is needed only for the full audit + overhaul plan (/seo-engine). Skip this entirely if you only want to run /seo-scan and /seo-qa — both are $0. There is no separate LLM or Anthropic key: the per-page AI relevancy and meta-rewrite pass runs through your own Claude Code session.

    To enable the paid audit later, create an account at api.dataforseo.com, then copy the credentials template and fill in your login / password:

    cp "Sales/_tools/keyword-data/.credentials.example.json" \
   "Sales/_tools/keyword-data/.credentials.json"
# then edit .credentials.json and fill in: login / password

    The credentials file is gitignored. Spend is always bounded by a --cap (USD, default 0.50) and a server-side ceiling of $2.00. Per-call cost rates in the source are unverified approximations — confirm against your live DataForSEO dashboard before authorizing a large run.

  4. Start the dashboard & log in

    From Code/SEO Audit/pipeline/, start the server:

    # from Code/SEO Audit/pipeline/
node server.mjs            # -> http://localhost:5505  (set SF_PORT to change)

    Open http://localhost:5505 and log in. Two accounts are seeded on first start, both with the password Default1:

    Seeded dashboard accounts and their scope
    Username Role Sees
    james superadmin Everything — all agencies, clients, sites, and scans.
    coty user Only what they own.
    Change the passwords before you expose it

    Both seeded accounts ship with the shared default password Default1. Change both passwords (or remove coty) before exposing the dashboard beyond localhost. A session-signing secret is auto-generated once on first boot at data/.session-secret. If port 5505 is taken, set SF_PORT to a free port.

  5. Run your first scan & see it in the dashboard

    Run a free scan. The <url> is required — never run it bare (a missing URL silently defaults to https://example.com). The --owner 1 / SF_OWNER_ID=1 tag makes james the owner so it shows up under the superadmin account: 

    # slash command
/seo-scan https://example-client.com --owner 1

# raw CLI equivalent (Bash / Git-Bash)
SF_OWNER_ID=1 node run.mjs https://example-client.com

# raw CLI equivalent (PowerShell)
$env:SF_OWNER_ID=1; node run.mjs https://example-client.com

    You will see three steps print, then it stops:

    [1/3] SCAN     free SF CLI, headless
[2/3] INGEST   INSERT a new crawl + urls/issues/images/links
[3/3] REPORT   writes data/scans/<crawlId>/index.html + health, refreshes data/index.html

    Now refresh the dashboard. The new crawl appears in the site list and in the domain trend, and its report is served at /scan/:id — the Screaming Frog report with a health score, the every-page × every-column grid with colour-flagged problem cells, and per-page detail accordions. This CLI run is byte-for-byte identical to clicking Run scan in the dashboard; both go through the same run.mjs runner.

    Don’t set SEO_DB

    SEO_DB is the isolated-test override. Leave it unset so the scan writes to the canonical data/seo.db — setting it would hide your scan from the dashboard.

Scan a site for free

Goal: Crawl any URL and get a Screaming Frog health report plus an every-page audit grid, ingested into the dashboard so it shows up in the site’s trend.

$0 Anyone Lightest entry point

Needs: Node.js ≥ 22.5 (Node 24 needs no flag) and the free Screaming Frog SEO Spider CLI installed with its EULA accepted once — launch the desktop app a single time or headless crawls refuse to run.

  1. Run the scan from the pipeline directory

    The <url> is required — never run it bare, because a missing URL silently defaults to https://example.com. Pick the line for your shell:

    /seo-scan https://example-client.com --owner 1
    SF_OWNER_ID=1 node run.mjs https://example-client.com
    $env:SF_OWNER_ID=1; node run.mjs https://example-client.com
  2. Watch three steps print, then it stops

    [1/3] SCAN (free SF CLI, headless) → [2/3] INGEST (inserts a new crawl plus urls / issues / images / links into the central data/seo.db) → [3/3] REPORT (writes the scan report and health, refreshes the home index).

  3. Open the report in the dashboard

    Start the dashboard and read the scan under /scan/:id:

    node server.mjs
# then open http://localhost:5505  (log in: james / Default1)

    You’ll see the Screaming Frog report (index.html) with a health score, the every-page × every-column grid with colour-flagged problem cells, and per-page detail accordions.

You end with: a completed crawl in data/seo.db and its report at data/scans/<crawlId>/index.html, visible in the dashboard and the site’s domain trend.

Gotchas

This is byte-for-byte identical to clicking Run scan in the dashboard — both go through run.mjs.

Do not set the SEO_DB env var. It is the isolated-test override; setting it hides the scan from the dashboard. Leave it unset so the scan writes to the canonical data/seo.db.

--owner / SF_OWNER_ID: 1 = james (superadmin, sees all), 2 = coty (sees own), unset = NULL (superadmin-only, listed “—”). Default to 1.

Goal: Produce an implementer-ready, multi-tab overhaul plan — per-page target phrase (SERP-backed: volume + KD, never CPC), competitor blueprints, on-page relevancy verdicts, meta rewrites, internal linking, schema, canonical, positioning, and a prioritized action plan.

Paid Operator optimizing a real client site

Needs: everything /seo-scan needs, plus DataForSEO credentials at Sales/_tools/keyword-data/.credentials.json (copy .credentials.example.json, fill in login / password). No separate LLM key — the per-page AI pass runs through your own Claude Code session via the Workflow harness.

Spend & the 3-Act shape

This is the only workflow that spends money: real DataForSEO SERP usage, hard-gated by --cap (USD, default 0.50). A large ecommerce catalog can take 30+ minutes of paid SERP (rate-limited ~12 req/min). Because a Node script cannot launch a Claude Code Workflow, it runs in 3 Acts around an approve-before-spend gate — Act 2 (the per-page AI pass) is launched by hand. The per-call rates in source are unverified approximations; confirm against your live DataForSEO dashboard before authorizing a large run.

  1. Act 1 — capture + $0 coverage gate (cheap, no paid SERP)

    One command builds the deterministic spine into an isolated per-site DB data/<slug>.db, writes a $0 coverage gate, and stops for your sign-off:

    /seo-engine https://example-client.com --cap 0.50
    node seo-engine.mjs https://example-client.com --cap 0.50

    It emits data/<slug>.llm-args.json and data/scans/<crawl>/coverage-review.html. Optional flags: --slug name, --skip-scan (reuse the existing DB), --sitemap (big programmatic sites).

  2. Gate — approve before spend

    Open coverage-review.html, tick the pages to fully optimize (money pages are pre-checked, support pages are opt-in), and download approvals.json. If money pages only is fine, skip the gate — Act 1’s defaults already cover them. To scope the paid fan-out to your approved set, re-invoke with the file:

    node seo-engine.mjs https://example-client.com --approvals path/to/approvals.json --cap 0.50
  3. Act 2 — launch the per-page Workflow (you do this)

    Read data/<slug>.llm-args.json, then launch the Workflow. The scriptPath must be an absolute path and args must be the actual JSON object, not a stringified blob:

    Workflow(
  scriptPath="<abs>/pipeline/llm-fanout.workflow.mjs",
  args=<full contents of data/<slug>.llm-args.json>
)

    One read-only agent runs per money page (Phase C on-page audit → Phase B meta rewrite). Save the returned {audits, metas} to _<slug>-llm.json.

  4. Act 3 — persist + regenerate (one command)

    Persists Phase C → relevancy_audit and Phase B → meta_rewrites, and regenerates the report with the On-Page Audit + Meta Rewrite tabs filled:

    node seo-engine-llm.mjs finish <slug>

You end with: Code/SEO Audit/pipeline/data/scans/<crawl>/optimization-preso.html — the multi-tab overhaul plan (Action Plan, Target Validation, On-Page Audit, Meta Rewrite, Internal Linking, Structured Data, Per-Page Plan, Pillar Constitution, Content Map, New-Page Opportunities).

Gotchas

Don’t expect one-click from the dashboard. The in-app “Approve & generate plan” button fires the same skill and fills schema / links, but the dashboard cannot launch the LLM fan-out (Act 2 is a Workflow; a node-spawned child can’t start one). An approve→spend run leaves meta_rewrites / relevancy_audit empty until you finish on the CLI with Act 2 + Act 3. The server fail-louds this as exit -28 rather than reporting a silent “done.”

Spend is bounded twice: your --cap (default 0.50) and a server-side ceiling PLAN_CAP_CEIL = 2.00 so a UI typo can’t overspend.

The engine fail-closes at the start if it can’t resolve a real brand or business type (a null brand / unknown business type aborts) — it refuses to emit a false-green plan. Confirm the crawl captured a real homepage.

Verify a plan actually landed

Goal: Prove what an optimization plan actually changed on the live site — did the prescribed titles / metas / H1 / schema land — and / or where the site was vs where it is now.

$0 Anyone verifying dev work Read-only proof, not a self-reported “done”

Needs: the central data/seo.db with two crawls in it (a plan crawl and a later re-scan). No Screaming Frog crawler needed. One live HTML fetch per page (for JSON-LD), with an on-disk snapshot fallback. Two modes run off one engine (qa-diff.mjs).

Mode A — PLAN→ACTUAL (“did the prescribed plan land?”)

  1. Re-scan the site after dev applies the fixes

    Run /seo-scan, click the dashboard Re-scan button, or mark the plan applied (which auto-fires a confirmation re-scan).

  2. Diff the plan crawl against the re-scan

    The numbers are crawl IDs:

    /seo-qa 13 22
# or:
node qa-compare.mjs <planCrawl> <rescanCrawl>
# writes data/scans/<rescan>/qa-diff-<plan>.json
  3. Or read it in the Command Center

    Open /pipeline/:cid/:sid, hit Recompute, and read the compare view.

Mode B — BEFORE→AFTER (plan-agnostic)

“Where was the site vs where is it now?”

/seo-qa --ba 13 23
# or:
node qa-compare.mjs --ba <beforeCrawl> <afterCrawl>
# writes data/scans/<after>/qa-ba-<before>.json
# in-app: the ?view=ba toggle on the Command Center

You end with: a QA diff JSON cache under the re-scan’s scans dir plus an in-app dark report. PLAN→ACTUAL verdicts per page: resolved / gap / changed_different / page_not_found / not_fetched.

Gotchas

Check slug alignment first. A plan slug absent from the re-scan is scored page_not_found, never silently “resolved.” If the matched / plan-slug % is near zero, keying failed — do not trust the verdicts.

Know what’s hard vs soft. Title, meta description, H1 focus, schema type, and schema breadcrumb are HARD pass/fail (title/meta/H1 read deterministically from each crawl’s Screaming Frog row; schema verified by parsing each live page’s JSON-LD). Subtopic coverage is a body-keyword HEURISTIC (reported, not scored). Internal links are ADVISORY.

qa-compare.mjs opens data/seo.db hardcoded read-only — no --db / SEO_DB override, so QA always runs against the central DB.

Control spend (approve before you pay)

Goal: Guarantee no paid work runs until you have reviewed a free coverage snapshot and approved the exact pages — and cap the dollar amount even if you fat-finger it.

$0 gate Paid phase Anyone authorizing a paid run

Needs: a scanned site, DataForSEO credentials (for the eventual paid step), and the dashboard running if you use the in-app gate. The gate itself is $0; only the post-approval paid phase spends, bounded by your cap.

In the dashboard

  1. Start the $0 coverage gate

    On a scan, POST /scan/:id/plan/start — starts the gate phase and parks the job at awaiting_approval. No paid work has run.

  2. Review coverage and tick pages

    GET /scan/:id/plan/coverage — review the gate and tick the pages you want optimized (money pages pre-checked, support opt-in).

  3. Approve to begin paid work

    POST /scan/:id/plan/approve — writes approvals.json and flips the job to the paid plan phase, scoped to your approved pages. Only now does paid work begin.

  4. Set your cap

    Use --cap <USD> on the CLI (default 0.50). A server-side clamp PLAN_CAP_CEIL = 2.00 sits over your cap so a UI typo can’t overspend.

On the CLI

  1. Run Act 1 — it stops at the gate 
    node seo-engine.mjs <url> --cap 0.50
# writes coverage-review.html and stops
  2. Approve, then scope the paid fan-out

    Tick pages, download approvals.json, then re-invoke with --approvals:

    node seo-engine.mjs <url> --approvals approvals.json --cap 0.50

    Unapproved money-candidates become support rows (held out of paid competition). A named-but-missing approvals file fails closed — it does not silently fall back to the full set.

You end with: an approvals.json scoping the paid run, and a plan job that only ever spends within your cap (and never above the $2.00 server ceiling).

Gotchas

The spend gate is a separate serialized queue from SF crawls, so a slow paid plan never blocks scanning and vice-versa.

Plan dev lifecycle: generated → passed_to_dev → applied. Marking a plan applied fires a confirmation re-scan of the same site automatically.

Cost rates in source are unverified approximations — confirm against your live DataForSEO dashboard before a large run.

Crawl 13 (the SGEN reference scan) is locked — plan-gen, deletion, and schema mutation are hard-blocked for it. Never target it.

Handle a big site (over the 500-URL cap)

Goal: Audit a site larger than the free Screaming Frog 500-URL-per-crawl cap by capturing from its sitemap instead of crawling.

$0 capture Paid optimize Large programmatic / ecommerce sites

Needs: a reachable sitemap (/sitemap.xml, /sitemap_index.xml, /wp-sitemap.xml, or /sitemap-index.xml) or a direct sitemap URL, and Node ≥ 22.5. Capture is $0; the subsequent /seo-engine optimization is paid as usual (and can be 30+ minutes of SERP on a large catalog).

  1. Recognize the cap

    A free-tier crawl stops at ~500 URLs. For bigger sites you must choose the sitemap path by hand — the auto-switch logic exists but is not wired into the live orchestrator, so pass --sitemap explicitly.

  2. Capture from the sitemap 
    /seo-engine <url> --sitemap
# or directly:
node sitemap-capture.mjs <url|sitemap-url> [--max N] [--slug s] [--fast]

    It resolves the canonical origin, probes the sitemap candidates (fail-closed if none found), expands index → sub-sitemaps → page URLs (one level of nesting), and dedupes.

  3. Understand the verify pass

    By default a verify pass fetches every URL (concurrency 3) and keeps only status 200 + not noindex + ≥150 words (--fast skips this). It is block-graceful: after repeated fetch errors it assumes a bot block, stops, and keeps the remaining URLs unverified rather than dropping them.

  4. Gate then optimize

    Use the $0 coverage gate to scope which pages to optimize — Stage 1 alone feeds the cheap coverage gate, so a 500-page site can be captured and gated with zero paid SERP. Then proceed through Acts 2–3 of the full-audit workflow.

You end with: a sitemap-captured crawl DB (default data/<slug>-sitemap.db) holding page rows, ready to gate and optimize.

Gotchas

Sitemap-captured pages carry no on-page data (cols_json is empty) — they are slug-classified downstream only. Any stage needing Title / H1 / word count gets empty fields for those pages.

Known residuals: no gzipped-sitemap decode, and only one level of index nesting. Auto-routing from the 500-cap is unwired — the operator must pass --sitemap.

How it works

  1. Scan

    The free Screaming Frog SEO Spider CLI crawls the URL headless and exports CSVs. The free edition caps at 500 URLs; for big programmatic sites a separate sitemap-capture path fetches and verifies every sitemap URL instead.

  2. Ingest

    The CSVs are parsed and written atomically into one central SQLite database (data/seo.db) across seven crawl tables. The full Screaming Frog row for each URL is preserved as JSON, so every later step reads Title, H1, and meta deterministically without re-fetching the page.

  3. Audit / Target

    The engine derives a site & business map, discovers the keyphrase universe, picks the right target phrase per page, builds competitor blueprints, and validates each target against its live SERP. This is the only paid layer (DataForSEO) — and a spend cap is checked before any spend.

  4. Phase B / C (+ AI)

    Deterministic Node prepares one input bundle per page, then a Claude Code Workflow — the single non-deterministic step, run from your own Claude session with read-only agents — writes the meta rewrites (Phase B) and on-page relevancy verdicts (Phase C). Node then persists the results and regenerates the report.

  5. Report

    A self-contained, multi-tab optimization-preso.html lands the results. Tabs that have no data graceful-empty rather than break — so a deterministic-only run (no AI step yet) still produces a valid report.

  6. Dashboard

    The dashboard on port 5505 serves the agency → client → site funnel, a serialized scan queue, an approve-before-spend plan gate, and an in-app renderer for the report artifacts.

  7. QA loop

    After you apply fixes, a re-scan + diff proves what changed — PLAN→ACTUAL (“did the plan land?”) and BEFORE→AFTER (“where was the site vs now?”), both read-only and $0, feeding an optional second-pass re-plan.

The tech stance

The whole system is built to stay free-tier, deterministic, and honest. It is pure Node.js (ESM, no transpile step) with zero npm dependencies — every module imports only Node built-ins, and there is no package.json to install. Data lives in a local SQLite database via Node’s built-in node:sqlite (which is why Node ≥ 22.5 is required), and the dashboard is a single zero-dependency node:http server on port 5505.

Crawling uses the free Screaming Frog SEO Spider CLI (the 500-URL cap is the only free-tier limit). The single paid input is DataForSEO, and only the full /seo-engine audit reaches for it. The per-page AI work spends your own Claude Code session tokens — there is no Anthropic API key anywhere in the path.

Why the AI step is shaped the way it is

A Node process can’t run an LLM here — there is no API key. So the per-page intelligence runs in a 3-Act shape: Node prepares the bundles (Act 1), you launch the Claude Workflow that does the actual writing (Act 2), and Node persists the result and regenerates the report (Act 3). The dashboard deliberately cannot do Act 2 on its own — it fails loud and tells you to run /seo-engine rather than emit an empty result silently.

Honest about cost

The DataForSEO per-call cost figures in the code are flagged unverified — confirm them against your live DataForSEO dashboard before authorizing spend. Running with --offline / --dry skips every paid stage and runs only the $0 layers.

Status & roadmap

Known issues — two live bugs on a fresh database

Both bugs are real and currently unfixed in the main engine (Code/SEO Audit). They were already fixed in the separate packaged copy at _dist/seo-dashboard-toolkit — so the fix is known; this is a port, not a new design. Neither fires on the owner’s populated local database (it already has the tables), which is why they have gone unnoticed. Both fire on a fresh/empty database, so they block any clean deployment or second operator. Proven by clean-room boot tests on 2026-06-16.

Bug 1 — the server cannot cold-start on a fresh database. Open
Trigger: launch the dashboard against a brand-new, empty database. Impact: startup throws no such table: crawls and the server never comes up. The cause is that the org schema builds performance indexes on the crawls/urls tables before the scan ingest creates those tables. Fix (already in the toolkit copy): create those indexes after ingest has made the tables, or guard them so they only build if the table exists.

Bug 2 — the command center returns HTTP 500 on a site that has no plan yet. Open
Trigger: open GET /pipeline/:cid/:sid for a site that has been added but for which no optimization plan has been generated. Impact: the page errors with no such table: meta_rewrites as an HTTP 500, because that table is only created during a paid plan run but the command-center query reads it unconditionally. Fix (already in the toolkit copy): tolerate a missing meta_rewrites table — create-if-not-exists, or guard the read with a left join.

Apart from these two cases, a clean-room boot proved the server cold-starts, seeds the two accounts (james superadmin, coty user; password Default1), and every other authenticated route renders 200 on an empty database.
