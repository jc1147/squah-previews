Analyze Site & Content — Site & Business Map (site_map.json)

done

Turn the ingested crawl into a deterministic, $0 site map — brand-stripped theme, per-page type/role/topics, the read offering (category/products/use-cases as noun phrases), served verticals, derived seed avenues, and a ranked money_pages list — that is the input foundation every later stage consumes.

What it accomplishes
  • Read every kind='page' row of the chosen crawl from the SQLite DB and extract the structural/textual signal the free Screaming Frog export actually stores (Title, H1-1/H1-2, H2-1/H2-2, Word Count, Crawl Depth, Inlinks, Indexability, Address) — never assume a body-text column that does not exist
  • Derive the BRAND and the host/preview PATH-NOISE first, so all topic analysis downstream is brand-free and host-path-free (crawl 13: brand='sgen' from the universal '| SGEN' title tail; path noise = squah/previews/sgen-site-v2/v2/github)
  • Classify every page into a TYPE (home/pricing/compare/product/solutions/resource/blog/about/contact/legal/utility/other) and a structural ROLE (pillar vs supporting) — driven by page type, since crawl-13 inlinks are nav-saturated and depth is degenerate (0/1)
  • Compute the whole-site THEME (document-frequency-aware term ranking) and per-page TOPIC_TERMS from the present textual signal (title+H1+H2+slug)
  • Read the OFFERING as multi-word NOUN PHRASES from the highest-intent pages (home boosted) — category + products + use_cases — never a single token and never the brand
  • Detect served VERTICALS/personas and build the derived SEED_AVENUES (self / offering / vertical) with no hand-typed seeds
  • Produce an opportunity pre-rank and a ranked MONEY_PAGES list (commercial-intent pages only; legal/policy + utility/contact/about excluded) — the slice that justifies the later paid R&D
  • Persist site_map.json to data/scans/<crawlId>/ as the single feed for Stages 2-6
The logic — how it's done
  1. LOAD (fail-closed gate): SELECT the crawl+domain (crawls JOIN sites); THROW if no crawl. SELECT all urls WHERE crawl_id=? AND kind='page'; THROW if 0 page rows. Parse each row's cols_json and map ONLY the columns the SF free export stores: address, Title 1, H1-1/H1-2 (first non-empty as h1), H2-1/H2-2, Word Count, Crawl Depth, Unique Inlinks||Inlinks, Indexability. (Verified live on crawl 13 = 59 page rows.)
  2. DERIVE BRAND (two unioned signals): (a) the registrable domain label (jc1147.github.io -> 'github', harmless dupe of path noise); (b) the most common trailing title segment after the last | / em-dash / en-dash / ' - ' separator, kept only if it dominates >=50% of pages-with-a-separator. crawl 13: '| SGEN' on 59/59 -> brand token 'sgen', display 'SGEN'. Brand tokens are excluded from every downstream tokenizer; the display string is kept on site_map.brand.
  3. DERIVE SHARED PATH-NOISE: compute the longest common leading URL path-segment prefix shared by >=80% of pages, tokenize each shared segment (splitting 'sgen-site-v2'), and record prefixLen. crawl 13 -> noise {squah,previews,sgen,site,v2} (+ github), prefixLen=2. The combined extra stop-set fed to every tokenizer = brandTokens UNION sharedPathNoise. Tokenizer base rule: lowercase, keep [a-z0-9 -], split, drop tokens <3 chars / pure-digit / in STOP / in PATH_NOISE / in extra.
  4. CLASSIFY PAGE TYPE per page via a priority cascade on the meaningful slug segments (URL path AFTER prefixLen) + title: depth<=0 or site-root index -> home; showcase/customers/reviews/case-study -> about; legal/privacy/terms/cookie/dpa -> legal; status/sitemap/404/login/changelog -> utility; compare/vs/alternative -> compare; pricing/plans/cost -> pricing; solutions/industries/use-cases/for- -> solutions; product/features/modules/platform -> product; resources/guides/docs/migrate -> resource; blog/news/articles -> blog; about/company/team -> about; contact/demo/sales -> contact; else other. A NESTED section index (e.g. /review/showcase/index.html) is deliberately NOT classified home (prefix-aware).
  5. COMPUTE STRUCTURAL ROLE: role='pillar' if depth<=0 OR type in {home,pricing,compare,product,solutions}, else 'supporting'. (Type carries the signal because inlinks are nav-saturated — 56/59 pages = 59 inlinks — and depth is degenerate 0/1; the old depth+inlink heuristic could not differentiate, so it is NOT used here.)
  6. COMPUTE PER-PAGE TOPIC_TERMS: weighted token frequency over the present signal — title x3, h1 x3, each h2 x2, slug-tokens x2 (last 1-2 slug segments, extra-stripped); take top 10. Set no_topic=true when zero terms survive (GATE: flagged, never dropped).
  7. COMPUTE WHOLE-SITE THEME: across all pages tally totalOccurrences + pageFrequency of each extra-stripped token (from title+h1+h2+slug); score = pageFrequency*2 + totalOccurrences (document-frequency-aware: a term on many pages outranks one repeated on a single page); keep top 25.
  8. READ THE OFFERING as multi-word noun phrases from HIGH-INTENT pages (home/pricing/product/compare/solutions). CATEGORY = top-scoring bigram/trigram across brand-stripped TITLES+H1s with a homepage weight of 4x (the home states the category), keeping only 'meaningful' n-grams (>=2 tokens, at least one non-generic); fall back to a site-theme bigram, then flag empty_category if nothing. PRODUCTS = top n-grams distinctive to product-type pages. USE_CASES = 'for <X>' phrase captures across all page text + n-grams on solutions pages. GATE: if no high-intent page exists, flag no_high_intent_page and fall back to theme rather than fabricate.
  9. DETECT VERTICALS: match a fixed industry/persona lexicon against (a) the offering use_cases (x2), (b) topic_terms of COMMERCIAL pages only — legal/utility excluded so a policy page's 'legal' token is not mistaken for a served vertical (x1), and (c) the slug of solutions pages (x3, strongest explicit signal); rank, take top 10.
  10. BUILD SEED_AVENUES (derived, never hand-typed): self = top 15 theme terms; offering = category + products + use_cases (deduped); vertical = each detected vertical crossed with the offering core (category + top products) as '<core> for <vertical>' and '<vertical> <category>'. All three are brand/path-noise-free by construction. These are the anchors Stage 2's THESIS builder consumes.
  11. OPPORTUNITY PRE-RANK + MONEY_PAGES: opportunityScore = TYPE_WEIGHT*10 + min(words/100,15) + max(0, 6-depth) — type/commercial-intent dominates; word-count and depth are tie-breakers; inlinks deliberately NOT used. page_rank = all pages by opportunity desc. money_pages = the opportunity-ranked pages EXCLUDING NON_MONEY_TYPES {legal,utility,contact,about} — i.e. only the commercial-intent pages worth the later paid funnel spend. (This is the fix for the old heuristic surfacing high-word-count LEGAL pages as 'top money pages'.)
  12. PERSIST: write site_map.json to data/scans/<crawlId>/ (opts.persist!==false; opts.scansDir lets the smoke test write to a temp dir so it never touches the live tree). The module is PURE: DB read + local text analysis only, NO DataForSEO/network calls, $0. Verified persisted artifact exists at pipeline/data/scans/13/site_map.json (generated_at 2026-06-10T18:06, page_count 59, brand SGEN).

Fail-closed gates
  • THROWS if the crawlId is not a positive integer
  • THROWS if no crawl with that id exists in the DB
  • THROWS if the crawl has 0 kind='page' rows (nothing to map — fail-closed, never emit an empty map)
  • FLAGS (never silently drops) any page with no extractable topic term (no_topic=true + a flags[] entry listing the URLs)
  • FLAGS an empty/unreadable offering: no_high_intent_page when no home/pricing/product/compare/solutions page exists (falls back to site theme, does not fabricate a category); empty_category when no phrase is extractable anywhere
  • Brand tokens and shared-path noise are stripped BEFORE any theme/offering/seed computation, so no brand-only or host-path 'topic' can leak into targets (preserves the locked brand-protection + relevance discipline)
  • money_pages structurally EXCLUDES legal/utility/contact/about so no policy/operational page can be sent into the paid R&D of later stages
  • Module is PURE / $0: DB read + local text analysis only, zero DataForSEO/network calls — no spend risk at Stage 1
Edge cases
  • Free-tier SF export stores NO body text — topic signal is derived from Title + H1(s) + H2(s) + URL slug only; this is a documented data reality, not an omission (so Stage 4 relevancy work compensates with the full local HTML body that IS available for 58 pages)
  • Inlinks are nav-saturated on crawl 13 (56/59 pages = 59 inlinks; the depth-0 home shows 1 because its inlinks landed on the duplicate /index.html) — so inlinks CANNOT differentiate money pages and are deliberately not used in ranking
  • Crawl depth is degenerate (only 0 and 1) — depth is a minor tie-breaker only; page TYPE carries the structural-role + money-page signal
  • GitHub-Pages preview host (/squah-previews/sgen-site-v2/) injects host/repo/preview segments into every URL — caught by both the fixed PATH_NOISE set AND the dynamic >=80% shared-prefix detector so they never become topic terms
  • Every title ends '| SGEN' -> brand reliably 'sgen'; the dominance rule (>=50% of pages-with-a-separator) prevents a one-off title tail from being mistaken for the brand
  • A nested section index (e.g. /review/showcase/index.html) must NOT be classified as the site-root home — the type classifier is prefix-aware to avoid this
  • The home page is marked Non-Indexable in crawl 13 (verified in the artifact) — Stage 1 records indexability verbatim and does not act on it; downstream stages decide
  • Single generic tokens (price/image/real/etc.) are kept OUT of the offering by the meaningful-n-gram filter, anticipating the later locked rule that single-token anchors are forbidden
Depends on: Depends only on a successfully ingested crawl in the DB (the crawl->CSV->SQLite layer that already exists). No other stage precedes it. It is the input foundation that Stages 2-6 all read: Stage 2 (Page Thesis + intent/winnability VALIDATE) consumes offering.category, money_pages[], verticals, per-page topic_terms/title/type, brand_tokens, and seed_avenues; later stages reuse the same artifact. Keep as-is — no changes proposed.

Page Thesis + Intent/Winnability VALIDATE (per-page focus gate: KEEP / RE-TARGET / FAIL-CLOSED)

build next

For each of the 47 money pages, establish what the page genuinely IS (its thesis) and adversarially validate its current _quickplan-picked focus against SERP-intent, winnability, and page-reality gates — emitting a per-page, evidence-backed verdict (KEEP / RE-TARGET / FAIL-CLOSED) so every downstream stage builds on a focus the page can actually win, not a high-volume slug term.

What it accomplishes
  • Catch the proven _quickplan failure class: focus chosen by highest search VOLUME on a slug-derived candidate with NO intent/winnability/page-reality gate (forms.html = 'online form builder' vol 9900 KD 71, 0x on page, free-tool SERP, paid $39/mo module = intent mismatch).
  • Establish a PAGE THESIS for all 47 pages from real signals (crawl title/H1/H2 outline + local HTML body/offer/CTAs/schema), de-poisoned of nav/footer chrome, stating what the page IS + the buyer/intent it serves — the anchor every gate measures against.
  • Run THREE adversarial gates per page: (G1) SERP-intent / page-type match from competitor blueprint rival-titles + related_searches vs the page type; (G2) winnability = KD viability for THIS page type (a KD-90+ head term on a single feature/solutions page is suspect); (G3) page-reality fit = does the page actually serve that query (presence + offer posture).
  • Emit one of three verdicts per page with evidence: KEEP (focus is right), RE-TARGET (a better intent-aligned focus + the gate evidence forcing the swap, optionally a wedge/comparison-page spin), or FAIL-CLOSED (no winnable, relevance-gated, intent-matched target exists).
  • Mark which pages CHANGED focus (RE-TARGET or FAIL-CLOSED) so Stage 3 re-derives competitor blueprints + content map ONLY for the changed set (~14 of 47), not all 47 — the persisted 47 blueprints + 80 content topics are correct for KEEP pages.
  • Persist verdicts durably (page_targets primary_phrase/intent/rationale + a verdict/evidence field) and produce the reviewable per-page HTML payload the owner approves before any Stage 3-6 execution — never mutate live site HTML.
  • Respect every locked constraint: SEO volume+KD only (never CPC), curated CORE-platform relevance gate as the hard backstop, brand-voice protection (never order keyword-stuffing a strong converting H1), fail-closed on ambiguity (flag/skip, never assert).
The logic — how it's done
  1. STEP 0 — LOAD + PRECONDITION (fail-closed): Read data/scans/13/site_map.json (offering.category='website platform', money_pages[] = the 49, verticals[]) and the 47 persisted page_targets (primary_phrase/volume/kd/intent + secondaries_json) + the 2 new_page_opportunities (already fail-closed) from seo.db via openDb(). For each money page resolve its url_id, page.type (home/pricing/compare/product/solutions/resource/blog/other), and local HTML path. Assert: 47 targets + 2 fail-closed = 49 money pages reconciles; any page with no resolvable focus AND no HTML → flag, never silently drop. The 2 already-fail-closed pages carry forward as FAIL-CLOSED (do not re-spend to re-confirm).
  2. STEP 1 — BUILD THE PAGE THESIS (per page, $0, deterministic + LLM-labeled): Parse the LOCAL HTML <main id='main'> only (verified on 58 pages); strip header/footer/nav/drawer/aria-hidden/script/style and any block recurring byte-identically on >=3 pages (the mega-menu naming 22 sibling features poisons body text — proven in Phase C spec). From the de-chromed main extract: the real H1, the full H1>H2>H3 outline (NOT the crawl's truncated H1x2/H2x2), the structural intro (first 1-2 <p> after <h1>), CTAs, pricing/offer posture, and schema.org type (e.g. forms ships SoftwareApplication+Offer => commercial/paid). Emit thesis = { what_it_is (one line from H1+offer), page_type, served_intent (transactional/commercial/informational/navigational, LLM-classified from the offer+headings, evidence-cited), offer_posture (free | freemium | paid-module $X | platform-tier), key_subtheme_nouns[] }. The thesis is the page's OWN truth — the yardstick all gates compare the focus against.
  3. STEP 2 — RE-DERIVE THE CANDIDATE SET (so RE-TARGET has somewhere to land, $0): Recompute the deterministic candidate anchors for the page via funnel.buildPageTarget(page, site_map) (slug+type-driven, multi-word-guarded, never single-token). This gives the same anchor family _quickplan started from PLUS the alternates _quickplan discarded when it sorted by volume-only (e.g. forms => 'website form builder' / 'form builder software' alongside 'online form builder'). The current focus is one member of this set; the gates decide whether it is the RIGHT member.
  4. STEP 3 — GATE G1: SERP-INTENT / PAGE-TYPE MATCH (the decisive gate; uses Stage-1 data, mostly $0). For the current focus, read its persisted competitor_blueprint (rival titles + PAA + related_searches) — already built for these focuses, so $0 for KEEP-candidates. Derive the SERP's DOMINANT intent: classify rival titles + related_searches (e.g. 'Free Online Forms', 'Google Forms', 'Tally', 'free unlimited', 'open source' => FREE-TOOL / informational-utility intent). Compare to the thesis page_type+offer_posture. MISMATCH triggers when SERP intent != page intent (forms: SERP=free-tool vs page=paid-module => MISMATCH) OR SERP rewards a different page TYPE (standalone tool vs embedded feature). Reuse the engine's classifyIntent() for deterministic wording-based intent + LLM judgment for the SERP-gestalt, every judgment evidence-cited (the exact rival titles/related terms). intent_mismatch is a HARD CEILING: a mismatched focus CANNOT be KEEP.
  5. STEP 4 — GATE G2: WINNABILITY (deterministic, volume+KD only — never CPC). winnability=(100-kd)/100. Apply a PAGE-TYPE-AWARE KD ceiling, not a flat one: a head term at KD>=90 is acceptable ONLY on home/category (where brand+breadth can compete and the page is exempt from exact-match), but SUSPECT on a single product/solutions/feature page (the live data exposes this: solutions/freelancers='freelancers website' KD 97, seo-optimization='seo for websites' KD 100 — unwinnable on a lone spoke). Rule: product/solutions/compare/resource focus with KD>=85 AND a lower-KD in-category alternate existing in the Step-2 candidate set => winnability FAIL => RE-TARGET to the alternate (rank alternates by volume x max(0.05, winnability), the engine's priority formula). KD>=85 with NO viable lower-KD alternate => FAIL-CLOSED. home/category KD-100 head terms (e.g. 'website builder' KD 100) are KEPT with a flag, never stuffed (brand-voice protection).
  6. STEP 5 — GATE G3: PAGE-REALITY FIT (deterministic presence + posture, $0). Using the de-chromed thesis text from Step 1, compute tiered presence of the focus: exact_match / head_term_match / token_overlap in title/H1/intro/headings/prose (NOT nav-label or footer hits). A focus that appears 0x in prose (forms: 'online form builder' 0x anywhere) is a reality-fit red flag — combined with a G1 mismatch it confirms the page does not serve that query. CRITICAL brand-voice guard: a 0x-in-H1 result must NEVER auto-prescribe inserting the focus into a strong converting H1 (forms H1='Build the form. Capture the lead. Route it anywhere.') — that is a forbidden regression; instead it is evidence FOR re-targeting to a phrase the page already embodies, or for a wedge page.
  7. STEP 6 — VERDICT SYNTHESIS (deterministic decision tree over the 3 gates, fail-closed): KEEP iff G1 match AND G2 winnable-for-type AND G3 served (focus in prose/title with intent-aligned posture). RE-TARGET iff a gate fails BUT a Step-2 alternate passes all three — emit the new focus + its volume/kd/intent + a rationale citing the exact triggering signal (e.g. forms => RE-TARGET to 'website form builder' / 'form builder software'; OR spin a /compare/online-form-builders wedge page when only a comparison framing wins the head term's intent). FAIL-CLOSED iff NO candidate in the set clears all gates (no demand, or every alternate is unwinnable/mismatched/relevance-failed) — record as a supporting/feature page with no rank target, exactly like the 2 existing fail-closed pages. When genuinely ambiguous, default to FAIL-CLOSED or flag-for-owner, never assert a KEEP.
  8. STEP 7 — RELEVANCE BACKSTOP ON ANY PROPOSED NEW FOCUS (the hard gate, may cost 1 SERP/page only for RE-TARGET candidates not already SERP'd). Any RE-TARGET focus must pass the curated CORE-platform gate before it is allowed: pull its SERP (or reuse cached serp_top10_json from phrase_research) and require serpDomainMatch >= 2 distinct CORE_PLATFORMS in top-10 (the ungameable-by-polysemy backstop). A proposed re-target whose SERP is not full of website-builder/CMS platforms is REJECTED back to FAIL-CLOSED — this prevents 'fixing' a bad focus with another off-category one. KEEP focuses already passed this gate when their blueprints were built; do not re-spend.
  9. STEP 8 — PERSIST + DIFF + REVIEWABLE PAYLOAD (no live-HTML mutation). Update page_targets in seo.db: for KEEP leave primary_* unchanged; for RE-TARGET overwrite primary_phrase/volume/kd/intent + set rationale + a new verdict field (KEEP/RE-TARGET/FAIL-CLOSED) and an evidence blob (thesis + 3 gate results + cited SERP/blueprint signals); for FAIL-CLOSED move it to new_page_opportunities with reason. Emit a changed_focus[] list = pages where primary_phrase changed or moved to fail-closed (the Stage-3 work order). Render the per-page verdict HTML payload (one card/page: thesis, current vs proposed focus, the 3 gate verdicts with evidence chips, the diagnosis line) for owner review — this is the reviewable artifact gating Stage 3-6 execution. Run-level reconciliation: assert every one of the 49 money pages lands in exactly one bucket {KEEP, RE-TARGET, FAIL-CLOSED} and counts sum to 49, else fail the run.
