Staging, in the box

Five one-click actions, all server-side.

Staging is not a workflow you assemble, it is a set of buttons. SGEN ships the staging copy, the two-way push, the clone, and the restore as native actions on every plan, each a single server-side click that runs as a real job and resolves to a confirmed state.

Push live to staging

Pull a fresh copy of your production site into the staging environment in one click, so you test against real content, not a stale fixture.

Push staging to live

Ship a tested change with one server-side button. The staging state replaces live as a point-in-time operation, with the files and database moved server-side, nothing to sync by hand.

Clone a site to another site

Stand up a brand new site as a full copy of this one in one click. SGEN copies the database, media, and settings and rewrites the URLs server-side, with no archive to download and re-install.

Restore from another site

Pull a different site's captured state into this site in one server-side click, so a known-good site can seed or rescue another.

Create a backup, then restore it

Capture the whole site, database plus every page, product, setting, and your full uploads folder, into one portable .sgen archive, and restore to that exact point on demand.

The engine underneath

Every push, clone, and restore runs off a point-in-time snapshot.

SGEN captures your entire site, a full database dump plus your complete uploads folder, into one portable, named, dated .sgen archive the moment you click Create a Backup. Each snapshot lands in the Backups panel as a point-in-time row you can rename, organize, and restore to that exact state. The staging push, the clone, and the cross-site restore all run off this same engine. See the Backups module, the portable .sgen archive, and manage every site from the site dashboard.

Questions

Frequently asked questions about Staging

How does push live and staging work? +
Each direction is one server-side click. Pull to Staging refreshes the test copy from production, Push to Live promotes the tested staging state to production. Both run as real point-in-time jobs off the snapshot engine, with the files and database moved server-side and nothing to sync by hand.
Is staging an extra cost? +
No. Staging is native on every plan, one of 23 modules, with no separate staging add-on and no upgrade tier to unlock it. Plans differ only by how many live sites you run. (Pricing as of June 2026.)
Can I clone a site, or restore from another site? +
Yes, each is one server-side click. Clone stands up a new site as a full copy of this one. Restore from another site pulls a different site's captured state into this one. Both run off the same point-in-time snapshot engine, with no archive to package and re-install and no URL rewrite to run by hand.
Are these snapshots automatic or off-site? +
No, and this is the honest limit: every snapshot is a manual, on-demand, same-disk point-in-time capture you trigger. There is no schedule, no automatic cadence, no off-site copy, and no integrity scan in this module. Fire one before a risky change, then push, clone, or restore from it.
Can I download a backup as a zip? +
No. Snapshots live in the panel as portable .sgen archives you restore, push, or clone from there, inside the platform. There is no self-serve download-a-zip button.
Is it on every plan? +
Yes. Staging, the two-way push, clone, and restore are all native, included on every plan. Paid plans start at $39/mo (billed annually; $47 monthly), and the free Sandbox lets you build and stage before you pay.
