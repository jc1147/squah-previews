The case for going plugin-free

A plugin is not a feature you bought once. It is a standing liability you rent.

A working WordPress site is rarely one product. It is a page builder, plus an SEO plugin, plus forms, plus security, plus caching, plus backups, plus redirects, plus custom fields, each its own license, its own update event, and its own third-party codebase running on the site your business depends on. That bundle charges three taxes, every month, forever.

The security tax

More plugins means more third-party code, and third-party code is where the vulnerabilities are reported: 91% of 2025's WordPress vulnerabilities were in plugins, per Patchstack. Every plugin you add widens exactly the surface those vulnerabilities live in, on a site you are liable for.

Plugin-free fix: no plugin layer means none of that reported surface is on your site to begin with.

The maintenance tax

Every plugin updates on its own schedule, independent of core and of every other plugin. Each update is a chance the page breaks, and each renewal is a chance you miss it and a feature quietly goes dark. Ten plugins is ten calendars and ten ways for a Tuesday-afternoon outage.

Plugin-free fix: native modules update with the platform, so there is no per-plugin compatibility roulette to babysit.

The cost tax

Each category above is a separate subscription with its own renewal date. The total depends on your stack, but the direction is always up, and none of it is the work your customers actually pay you for. It is overhead you carry just to keep the lights on.

Plugin-free fix: the categories ship as native modules on every plan, so the count of things to license does not climb as your site grows.

This is not an argument that WordPress cannot do these things. It is an argument about what it costs you in licenses, updates, and reported vulnerability surface to add them back, plugin by plugin.

The case for going plugin-free · common questions

The questions people ask before going plugin-free.

Is the 91% figure real, and where is it from? +
It is from Patchstack's State of WordPress Security 2026 report, which catalogues new WordPress vulnerabilities each year. Across 2025, the large majority of reported vulnerabilities were attributed to plugins rather than to core or to themes, and the report puts the plugin share at roughly 91%. The figure has sat in the same band for years (about 96% in the prior reporting), so the exact number moves a few points but the conclusion is stable: the plugin layer is where the vulnerabilities are.
Does removing plugins actually make a site more secure? +
The data is the argument. If 91% of reported WordPress vulnerabilities live in the plugin layer, then a platform with no plugin layer does not carry that reported surface in the first place. SGEN removes the plugin layer entirely and ships a WAF plus ongoing security patching free in the Foundation Pack on top of that. We are not claiming SGEN is unbreakable; we are saying the largest single category of WordPress risk is structurally absent.
Can I still add my own third-party plugins to SGEN? +
SGEN is not WordPress with a plugin directory; it is a native platform where the features you would normally reach for a plugin to add are already built in. Where you need custom HTML, CSS, or JavaScript, the Custom Codes module handles head and body snippets natively. So the one capability people fear losing when they leave the plugin world, adding custom code, is built in, even though there is no third-party plugin marketplace.
What if SGEN does not have a module for something I rely on? +
Start with the modules overview. The 23 native modules cover the categories most sites use a plugin for, and Custom Codes covers custom snippets. If a capability you depend on is genuinely not covered, that is a real question to raise with sales before you switch, rather than something we will hand-wave. We would rather tell you up front than promise parity we have not verified.
Is the native replacement as capable as the specific plugin I use today? +
That depends on the module and on which plugin you are comparing it to, so we do not make a blanket "as good as X" claim. Open the page for the module that interests you on the modules overview, see exactly what it does, and judge it against what you use now. We would rather you check the module than take a marketing claim on faith.
Will my site break when SGEN ships a platform update? +
The reason WordPress sites break on update is the plugin dependency tree, plugins updating on their own schedules, independent of core and of each other. There is no such tree on SGEN, because the features that would be plugins are native and update with the platform. When SGEN updates, your content and your design are not affected: there is no plugin layer to fall out of sync.
