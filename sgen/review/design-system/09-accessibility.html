Page 09 — operable by keyboard alone

Accessibility, proven — not promised.

Every widget on this page works with a keyboard only and reports itself correctly to assistive technology. Press Tab right now — the first stop is a Skip to main content link, then the focus ring lands on every interactive control in a logical order. Menus, tabs, an accordion and a modal each honour the WAI-ARIA keyboard contract (↑ ↓ ← →, Home/End, Enter/Space, Esc). This is the bar every SGEN block must clear.

Demo 4 · ARIA tablist with arrow-key selection

4Keyboard-operable tabsAPG · Tabs

A proper role="tablist": only the active tab is in the tab order (roving tabindex). ←/→ move between tabs and select on move; Home/End jump to the first/last; Tab from the tablist lands in the active panel. Each panel is wired with aria-labelledby back to its tab.

TryTab to the selected tab, then press → / ← to switch panels without touching the mouse.

Perceivable

Text alternatives for non-text content, captions, sufficient colour contrast, and content that doesn't rely on colour alone. Information must be presentable in ways all users can perceive.

Operable

All functionality available from a keyboard, no keyboard traps, enough time, no seizure-inducing flashes, and clear navigation. This whole page is the Operable principle, demonstrated.

Understandable

Readable text, predictable behaviour, and input assistance — labels, instructions, and error messages that tell the user how to fix the problem (see the form below).

Robust

Markup that parses cleanly and works with current and future assistive technologies — semantic HTML first, ARIA only to fill the gaps, name/role/value exposed on every control.

Demo 6 · the focus-trap test

6Keyboard-operable modal dialogAPG · Modal Dialog

The hardest a11y widget. On open, focus moves into the dialog; Tab and Shift+Tab are trapped inside it (you can never tab out to the page behind); Esc or the close button dismisses it; and focus returns to the button that opened it. It carries role="dialog", aria-modal="true" and an aria-labelledby title.

TryPress Enter on "Open dialog", then Tab repeatedly — focus cycles only inside the dialog. Press Esc; focus snaps back to the open button.

7ARIA patterns & the first rule of ARIA

The first rule of ARIA is don't use ARIA — prefer a native element that already has the role, name and keyboard behaviour. Reach for ARIA only to fill a genuine gap. Each widget above maps to one APG pattern; here's the contract each one upholds.

8Colour-contrast pairs — pass & failWCAG 1.4.3 · Contrast (Minimum)

Normal text needs a contrast ratio of ≥ 4.5:1; large text (≥ 24px, or ≥ 19px bold) needs ≥ 3:1. Each swatch below shows the computed ratio and a worded verdict — so the pass/fail signal never depends on the colour itself. The failing pairs are here on purpose, labelled FAIL, to show what to avoid.

Verdict thresholds: PASS ≥ 4.5:1 · LARGE-ONLY 3–4.5:1 (large text only) · FAIL < 3:1. This is why SGEN ships --accent-ink #a50f19 for red text on light, not the brighter --accent #d51522.

Demo 9 · respecting a setting, not removing motion

9Reduced motionWCAG 2.3.3 · Animation from Interactions

Some users get motion sickness from movement. The whole site is wrapped in @media (prefers-reduced-motion: reduce) which drops non-essential animation and smooth scrolling when the OS setting is on. Function is never removed — only the movement. Toggle it here to see both states without changing your OS.

Demo 10 · labels, errors & announcements

10Accessible form — labels & error statesWCAG 3.3.1/3.3.2 · Labels & Error ID

Every field has a real <label for> (placeholders are never a label). Help text is tied with aria-describedby. On a failed submit, each bad field gets aria-invalid="true", its error message is linked the same way, focus moves to the first error, and a role="alert" summary is announced. Errors are worded — they say how to fix it — and never rely on red alone.

TryPress Submit with empty fields: errors appear, focus lands on the first bad field, and a summary is announced. Fill them and submit again for the success state.

11The accessibility checklist

No block — hero, card, form, nav, modal — ships until it passes all of these. It's the same contract demonstrated live above, written as a gate.

Ships only when all 12 are true

Run top-to-bottom on every component before it leaves the styleguide. A single failure is a blocker, not a nit.

  • Keyboard-only operableEvery action reachable & usable with no mouse — no keyboard traps.
  • Visible focus on everything:focus-visible ring, ≥3px, contrasting on its ground.
  • Logical tab orderDOM order = reading order; skip-link first; no positive tabindex.
  • Semantic HTML firstNative button/a/label before any ARIA.
  • Name, role, value exposedEvery control reports itself to AT — APG pattern upheld.
  • Text contrast ≥ 4.5:1(≥3:1 for large text) — measured, not eyeballed.
  • Meaning beyond colourIcons / text reinforce every colour-coded state.
  • Labelled inputs + linked errorslabel for, aria-describedby, aria-invalid, focus to first error.
  • Modals trap & restore focusFocus enters, Tab trapped, Esc closes, focus returns to opener.
  • Honours reduced-motionNon-essential animation drops under the media query.
  • Targets ≥ 44×44pxTouch & click targets meet the minimum size.
  • Live regions announce changeAsync updates spoken via role="status" / alert.
