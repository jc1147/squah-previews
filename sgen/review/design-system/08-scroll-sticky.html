Page 08 — behavior in time

Scroll & sticky, live.

This is the see-it-work page. Eight scroll behaviors, each demonstrated as you scroll this very page — a static capture proves none of it. The header above is demo #1: right now it sits transparent over this dark hero; scroll a little and it goes solid glass, shrinks 84→58 px, and drops a shadow. Watch the chip in the header flip from at-top to is-stuck.

Scroll down — every behavior is labelled trigger → action → threshold.
The universal shape · library §1

Every sticky asset is one five-field machine.

Author each behavior by filling these five fields — nothing more. Pick its trigger family first (it dictates the primitive), then one boolean, a measured threshold, an inverse teardown, and the per-breakpoint variance. Every demo below is labelled in exactly this vocabulary.

FieldAnswersVocabulary
TriggerWhat scroll fact flips the state?one of four families — F1 scroll-position · F2 element-in-view · F3 scroll-direction · F4 non-scroll intent
ActionWhat changes when it fires?toggle ONE class / data-state; CSS does the rest (pin / shrink / slide / reveal)
ThresholdAt exactly what value?a measured px / ratio — header height, sentinel crossing, viewport fraction, depth % — recomputed on resize, never a magic number
TeardownWhen + how does it release?the inverse condition → remove the class, restore rest state, collapse reserved space
BreakpointDoes the rule change per width?enable/disable per tier, retune tolerances, relocate (corner FAB → bottom bar)
Demo 1 · the flagship

0Sticky header — pin + shrink + transparent→solidF1 · scroll-position

The header at the very top of this page is the demo. One boolean (is-stuck) drives pin, shrink and the transparent→solid flip together — exactly as BeTheme's mfnSticky() does. "At rest" is simply the flag's absence (at-top).

TriggerF1 — scrollY > headerH (not direction-aware)
Actionadd is-stuck → pin, shrink 84→58px, transparent→glass, shadow, border
Thresholdmeasured header height, recomputed on resize — never hardcoded
TeardownscrollY ≤ headerH → remove flag, restore transparent rest state

header: at-top · transparent

The anti-jump contract is load-bearing: a header that leaves the flow lets the page lurch up by its height. position:sticky reserves that space for free; a fixed header needs a spacer or padding-top. Transparent-over-hero only works with a dark/imagery hero and light nav text behind it — and must flip solid on scroll or the text becomes unreadable.

Scroll up and down past the hero edge and watch the header chip flip — that boundary is the measured headerH, not a guessed number.

Demo 2 · the Headroom model

2Smart-nav — hide on scroll-down, reveal on scroll-upF3 · scroll-direction

The white section bar directly under the header. Scroll down and it slides up out of the way (reclaims reading band); scroll up — even a little — and it snaps back instantly. The only family that must read the previous scroll position. The dir chip on the bar updates live.

TriggerF3 — compare scrollY to last each frame; down = hide, up = reveal
ActiontranslateY(-130%) to hide / translateY(0) to reveal — transform only
Thresholdoffset = header height · tol.down ≈ 6px (kill jitter) · tol.up = 0 (instant)
Teardownwithin offset of top → force-reveal; frozen while any overlay is open

smart-nav: pinned

It hides with transform only — never display:none — so a keyboard or screen-reader user keeps it in the tab order while it's translated off-screen, and focusing any nav link force-pins it back. Reveal is set to instant (tolerance.up = 0) because the moment a user reaches back toward navigation, they want it now.

Demo 3 · the primary conversion lever

3Sticky conversion CTA — appears past the hero, dismissableF2 · element-in-view

The charcoal bar pinned to the bottom of your viewport. It was hidden over the hero; once you scrolled past the hero's bottom edge it slid up. It carries a real × dismiss — close it and it stays gone for the session. A second observer would also hide it when the real in-page CTA scrolls into view, so it never doubles up on the actual buy button.

TriggerF2 — a sentinel at the hero's bottom edge exits the viewport
Actiontoggle is-visible → slide up translateY(115%)→0 + fade
Threshold"past the hero" via IntersectionObserver, biased earlier with rootMargin
Teardownscroll back into hero → hide · click × → dismissed for the session

sticky CTA: hidden (in hero)

Reveal is visual only — focus is never moved into the bar (jarring mid-read), and visibility:hidden while hidden keeps a tab-but-can't-see trap from forming. The CTA button is a full 44px tall touch target. Bottom placement sits in the thumb zone and clears a mobile address bar.

Demo 4 · the scroll utility

4Back-to-top FAB — appears past a committed-scroll thresholdF1 · scroll-position

The round red button bottom-right. It stays hidden until you've committed to scrolling — past roughly two viewport heights — then fades in. Click it and the page smooth-scrolls to 0 and moves focus to the top so nothing is stranded. It sits above the sticky CTA bar via the offset cascade, never overlapping it.

TriggerF1 — scrollY > max(1000, innerHeight × 2)
Actiontoggle is-visible → fade + scale in; click → scrollTo({top:0})
Threshold"committed to scrolling" — without cluttering a short page
TeardownscrollY ≤ threshold → fade out + remove from tab order

back-to-top: hidden

It's a real <button type="button" aria-label="Back to top">, not a styled <div>, at the full WCAG 44×44 tap size. Its bottom reads calc(var(--bottom-stack) + 18px) so it always clears whatever bottom bar is present — a structural fix, not a magic z-number.

Demo 5 · shown inline in a phone frame

6Mobile bottom bar — safe-area-aware call/cart barF2 + F3

On a real phone this bar is fixed to the bottom of the device. To prove it without a touch device, it's rendered inside a phone frame you can scroll right here — scroll the phone's content and watch its bottom bar slide up once you pass the hero, then hide again at the top.

TriggerF2 (hero sentinel exits) or F3 (past ~0.8 viewport & scrolling up)
Actiondata-state="shown" → translateY(0); reserve bottom padding
ThresholdSHOW_AFTER = innerHeight × 0.8 (past the hero)
Teardownback in hero / scrolling down → hide · hide on input focus (keyboard) · suppress while drawer open

Design Lab

Scroll this little screen. The bottom bar is hidden while you're at the top (the "hero"), then slides in once you pass ~80% of a viewport.

The real version is safe-area-inset aware (env(safe-area-inset-bottom) with viewport-fit=cover) so the buy button never hides behind the home indicator.

It hides itself when a text input is focused so it never fights the on-screen keyboard, and suppresses while the drawer is open so it can't cover the dismiss control.

Keep scrolling — the bar reserves body{padding-bottom} so it never covers the footer. On desktop it collapses to a corner pill instead.

That's the whole behavior: position-gated reveal, thumb-zone placement, keyboard coexistence, reserved space.

Demo 6 · layout, not motion

7Pin / release — sticky sidebar within a sectionCSS position:sticky

The dark card on the left pins as you scroll this section, then releases itself the moment the section's bottom passes it — pure CSS position:sticky, zero JavaScript, self-releasing. Its lifetime is the section's scroll length, so there are no thresholds to wire by hand.

TriggerCSS — section top reaches the top offset
Actionelement pins at top:calc(headerH + 22px) — clears the sticky header
Thresholdnone to wire — pin-start = section top hits top; release = section bottom passes
Teardownsection bottom passes the pin line → naturally releases · un-stuck on stack ≤767
PINNED · CSS sticky

I stay put while the column beside me scrolls.

No JS, no scroll handler. I pin when this section reaches the top offset and let go the instant its bottom edge passes me. Watch my top edge hold steady as the cards on the right move.

top: calc(headerH + 22px)

The #1 sticky bug

Sticky silently degrades to relative if no non-auto inset (top) is set, OR any ancestor up to the scroll root has overflow:hidden/auto or a fixed height.

align-items:start is required

The grid default stretch forces the media to full column height — it then has nowhere to travel. start lets it pin and move.

The top offset clears the header

Set top to the sticky-header height plus breathing room so a focused sidebar link isn't hidden under the bar.

Layout, not motion

Because pin/release is layout (not animation), no reduced-motion exception is needed — but it must still un-stick to a single column when the layout stacks on mobile.

No thresholds to guess

The pin lifetime is the section's own scroll length. You wire nothing — the browser computes the stick/release boundaries from the box geometry.

Reach for JS only when taller than the viewport

A sidebar taller than the viewport needs the scroll-then-pin-bottom pattern. Anything ≤ viewport is pure CSS.

Demo 7 · fires once, at depth

8Scroll-depth trigger — a banner that appears at ~60% depthF2 · element-in-view

A 1px sentinel sits at roughly 60% of the page's scroll depth. The instant it enters the viewport, the dark banner near the bottom slides in — once per session (it sets a flag and disconnects its observer). It's the higher-value mobile path versus exit-intent (phones have no mouseleave). Dismissable, and it never re-fires.

TriggerF2 — a sentinel at ~60% article depth enters the viewport
Actionslide the banner in; set the sessionStorage flag; io.disconnect()
Threshold~60% of scroll depth (tune 50–70%)
Teardownfires once per session · click × to dismiss · shares one flag with exit-intent

depth trigger: armed (not yet fired)

In a full implementation this and an exit-intent popup share one session flag, so a visitor sees at most one. Both must be real dialogs — role="dialog", focus-trapped, Esc-dismissible, with a labelled ≥44px close — and never full-screen on a first mobile view (the SEO interstitial penalty). Here it's a non-blocking banner so it doesn't trap the styleguide reader.

Everything on this page traces to a measured threshold, throttled with rAF + a ticking boolean, or observed off-thread with IntersectionObserver — never a guessed magic number.

How they coexist · library §3

Orchestrated structurally, not numerically.

Every sticky asset on this page is a direct child of <body> — a sibling in the root stacking context, never nested in a transform/opacity/filter ancestor that would swallow it. One named z-scale lives in :root; the offset cascade (--top-stack / --bottom-stack) drives every bar's edge so no bar hard-codes another's height. The back-to-top above literally clears the sticky CTA via that cascade.
