Page 11 — build to tokens, not hex

Theme by token, not by hand.

A site themed properly is themed in one place. Every surface, ink, line and shadow reads from a CSS custom property — so a whole site flips light↔dark, or re-skins to a new accent, by changing values, never markup. Below: a live sample site (header + hero + cards + footer) that you re-theme with the controls. Nothing in it carries a hard-coded color.

The chrome · five custom properties

1The design tokens — accent, ink, paper, shadow, radius.

This is the whole system. Five families of custom property declared once in :root. Every component on every page references them — so the brand lives in one block, and a re-skin is a re-set of these values, not a search-and-replace across the codebase.

The one rule

3Build to tokens, not hard-coded hex.

A literal hex in a component is a theme leak — it can't follow the light/dark flip and it can't re-skin. The fix is mechanical: a color in a rule is always a var() reference, and the literal only ever appears once, in the token declaration. The left column can't theme. The right column does, for free.

✗ hard-coded — a theme leakcan't flip, can't re-skin
/* color baked into the component */
.card{
  background:#ffffff;
  color:#1a1a1a;
  border:1px solid #e6e6e6;
  box-shadow:0 15px 30px rgba(1,7,39,.13);
}
.cta{ background:#d51522; }
/* dark mode? rewrite every rule. */
✓ tokenized — themes itselfflip + re-skin = free
/* color lives in the token layer only */
.card{
  background:var(--s-surface);
  color:var(--s-ink);
  border:1px solid var(--s-line);
  box-shadow:var(--s-shadow);
}
.cta{ background:var(--s-accent); }
/* dark mode = re-set the tokens. done. */
STEP 1

Declare on a scope

Light values on :root (or the stage root). Every token is named for its role (surface, ink, accent), not its color.

STEP 2

Override under a flag

[data-theme="dark"] re-sets the same token values. The skins re-set only the accent family. Structure is untouched.

STEP 3

Reference, never inline

Components only ever say var(--token). The cascade does the theming — flip the flag and the whole subtree re-paints.

Dark-mode done right

Dark mode is not "invert the colors."

A proper dark theme re-tunes the token values with intent — softer ink so it doesn't glare, raised surfaces instead of pure black, a tinted shadow that still reads on a dark ground, and an accent that holds its contrast. Same five tokens, re-valued for the dark surround.

surface

Raise, don't blacken

Dark surfaces sit a step above the page ground (#191b20 on #0e0f12) so cards still read as lifted — never pure #000.

ink

Soften the text

Body ink is near-white, not #fff — full white on dark vibrates. Dial it to a calm light grey for long reads.

accent

Hold the contrast

The accent must clear contrast on both grounds. A skin that passes on paper can fail on slate — verify on each.

shadow

Deeper, still tinted

Shadows go heavier and darker on a dark ground, but stay slightly ink-tinted so depth survives where black-on-black would vanish.
